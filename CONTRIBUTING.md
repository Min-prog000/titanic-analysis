# Contributing Guidelines

## Development Flow

- The `main` branch must always remain deployable.
- Create a `feature/<name>` branch for your work.
- Open a Pull Request (PR), get it reviewed, and merge after approval.

## Branch Naming Rules

|type|description|
|---|---|
|chore|Changes that do not affect production (e.g., task files, configs)|
|docs|Documentation updates|
|feat|Addition or modification of user-facing features|
|fix|Bug fixes that affect users|
|refactor|Code changes intended to improve structure without altering functionality|
|style|Formatting or style-related changes|
|test|Addition or modification of test code|

## Commit Messages

- Provide a one-line summary, e.g., `[feat] (form) :sparkles: create post form (#1)`.
- Add details in the body if necessary.

## Pull Request Procedure

- Use a clear and descriptive title (Japanese or English is acceptable).
- Follow the PR template.
- Ensure CI checks pass before requesting review.

## Code of Conduct

- Be respectful and constructive in discussions.
- Follow project coding standards and style guidelines.

## Testing

- Add or update tests when introducing new features or fixing bugs.
- Ensure all tests pass locally before pushing changes.

## Documentation

- Update `README.md` or other relevant docs if your changes affect usage or setup.
- Keep documentation bilingual (English + Japanese) when possible.
